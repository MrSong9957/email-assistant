# Email Assistant 产品化设计

## Context

邮箱助手作为 Claude Code Skill 已功能完整（IMAP/SMTP 多 Provider、AI 摘要分类、回复/转发/归档），在 CLI 中可正常使用。现在要将其从「个人工具」升级为「开源展示项目」，通过 Build in Public 建立个人品牌，为后续可能的框架化（方案 B）或 SaaS 化（方案 C）打基础。

## 目标

- **首要**：学习 AI 应用开发与开源运营的完整流程
- **次要**：在 AI 开发者 + 效率工具爱好者圈建立个人品牌
- **远期**：如果发现市场信号，探索直接变现路径

## 目标受众

- AI 应用开发者（使用 Claude Code / AI Agent 构建工具的人）
- 效率工具爱好者（终端工作流、键盘流、自动化）
- 两个群体高度重叠，活跃在 GitHub、Twitter/X、即刻

## 三阶段路线

### 方案 A（立即执行）：开源项目 + Build in Public

邮件助手本身不卖钱，「一个开发者用 AI 从零打造实用工具」的过程才是价值。

### 方案 B（有信号后启动）：AI Skill 开发框架

从邮件助手提取通用 Skill 骨架，做成 Claude Code Skill 模板/脚手架，让其他开发者用同样模式构建日历助手、RSS 助手等。

### 方案 C（市场验证后考虑）：AI 邮件助手 SaaS

Web 产品，多用户多邮箱，AI 原生邮件管理。仅在 A+B 阶段发现明确付费需求时推进。

本文档聚焦方案 A 的执行设计。

## 方案 A 详细设计

### 一、项目打磨

#### README 重写

- 一句话描述 + ASCII 架构图（用户指令 → Claude Code → Skill → IMAP/SMTP → 邮箱）
- Quick Start：3 步跑起来
  1. `pip install -r requirements.txt`
  2. 配置 `.env`
  3. 运行 `python email_cli.py fetch`
- 终端截图/GIF：展示邮件汇报表格的实际效果
- 技术亮点说明：多 Provider 适配、HTML 转发保留格式、Skill 架构

#### 安装体验优化

- `requirements.txt` 确保一键安装
- 现有 `install.sh` / `install.ps1` 打磨可用
- Docker 一键启动保持可用

#### 项目标配

- MIT LICENSE
- CHANGELOG.md

### 二、内容策略

#### 核心定位

内容主角是 AI + 开发过程，邮件助手只是载体。不做产品宣传，做开发日记。重点展示「遇到问题 → 分析 → 解决」的思维过程。

#### 平台与形式

| 平台 | 内容形式 | 目的 |
|------|---------|------|
| GitHub | 项目本身、Wiki、Issue 讨论 | 技术 credibility 的根基 |
| 即刻 / Twitter(X) | 短帖，开发进展、踩坑瞬间 | 圈子曝光、互动 |
| 抖音 / B站 | 2-5 分钟短视频，录屏+讲解 | 触达更广的技术受众 |
| 公众号 / 博客（可选） | 长文，设计决策深度解析 | SEO 沉淀、深度信任 |

#### 内容节奏（前 2 个月集中期）

- 每周 1-2 条短帖：GitHub star 里程碑、解决有趣的 bug、AI 代码 review 对比
- 每 2 周 1 个视频：功能演示、安装教程、从零实现某功能的录屏
- 1 个月 1 篇长文：如「我用 Claude Code 做了一个邮件助手，以下是学到的事」

#### 视频制作要点

- 开头 10 秒必须有 hook：「我用 AI 写了一个能帮我管邮件的终端工具」
- 录屏为主，配字幕和简要讲解
- 不追求高制作质量，真实感 > 精致感
